# Commit Message

Write a Git commit message for the currently staged changes. The **intent** of
the change is the most important element to capture.

## Workflow

1. Run `git status --short` to check for unstaged and untracked changes
2. If there are unstaged modifications or untracked files, stage them with
   `git add` (use specific file paths, not `-A`)
3. Run `git diff --cached` to inspect the staged changes
4. Run `git diff --cached --stat` for a file-level summary
5. If nothing is staged after steps 1-4, inform the user there is nothing to
   commit and stop
6. Determine the **intent** -- answer "why does this change exist?" in one
   sentence before writing anything. If the answer is not clear from the
   code alone, use AskUserQuestion to ask the user to clarify the purpose.
7. Write a commit message following the format below. The body MUST lead
   with the intent from step 6 -- the "why" sentence. Then run
   `git commit` with that exact message. Do NOT output the message and
   stop -- always follow through with the commit.

## Format

```
Capitalized, short (50 chars or less) summary

More detailed explanatory text, if necessary. Wrap it to about 72
characters or so. In some contexts, the first line is treated as the
subject of an email and the rest of the text as the body. The blank
line separating the summary from the body is critical (unless you omit
the body entirely); tools like rebase can get confused if you run the
two together.

Write the commit message in the imperative: "Fix bug" and not
"Fixed bug" or "Fixes bug." This convention matches up with commit
messages generated by commands like git merge and git revert.

Further paragraphs come after blank lines.

- Bullet points are okay, too

- Typically a hyphen or asterisk is used for the bullet, followed by a
  single space, with blank lines in between, but conventions vary here

- Use a hanging indent
```

## Rules

- The summary line MUST be 50 characters or less, capitalized, imperative mood,
  no trailing period
- Wrap body text at 72 characters
- The body MUST explain **why** the change exists, not just list what changed.
  If the body reads like a changelog or file inventory, rewrite it.
- Keep it concise -- omit the body entirely if the summary alone is sufficient
  for a trivial change
- Output only the commit message text, nothing else
- Do not include how the change was tested
- Do not include Co-Authored-By lines

## Bad vs Good

Bad -- restates the diff without intent:

```
Add commit and PR slash commands

- Add /commit command that invokes the commit-message skill
- Add /pr command to automate PR creation against staging
- Add commit-message skill with format rules and workflow
- Gitignore .claude/worktrees/ directory
```

Good -- leads with why, then summarizes what:

```
Add commit and PR slash commands

Standardize commit and PR workflows into repeatable commands
so messages follow consistent format rules and PRs are always
created against staging with the right reviewers.
```
