    1. In `internal/config/hosts.go`, modify `ActiveHost` to add a config.yml fallback step between `h.DefaultHost` and the hardcoded `defaultHost`:

    ```go
    func (h *HostsConfig) ActiveHost(flagHost, envHost string) string {
        if flagHost != "" {
            return flagHost
        }
        if envHost != "" {
            return envHost
        }
        if h.DefaultHost != "" {
            return h.DefaultHost
        }
        // Fall back to config.yml's default_host
        if cfg, err := ReadConfig(); err == nil && cfg.DefaultHost != "" {
            return cfg.DefaultHost
        }
        return defaultHost
    }
    ```

    Note: `ReadConfig()` is in the same package (`internal/config`), so no import needed. When `ReadConfig()` errors or returns empty `DefaultHost`, fall through to the hardcoded default. `DefaultConfig()` returns `DefaultHost: "app.keeperhub.io"`, but `ReadConfig()` only returns `DefaultConfig()` when the file is missing -- if the file exists but has no `default_host` key, the field will be empty string. This is correct behavior: we only want to use the config.yml fallback when the user explicitly set `kh config set default_host X`.

    However, there is a subtlety: `ReadConfig()` returns `DefaultConfig()` (which has `DefaultHost: "app.keeperhub.io"`) when config.yml does not exist. This means the fallback would always return "app.keeperhub.io" from config before reaching the hardcoded default -- which is the same value, so functionally correct. BUT if a user creates an empty config.yml (no default_host key), `ReadConfig` returns `Config{DefaultHost: ""}`, so the fallback correctly skips it. The check `cfg.DefaultHost != ""` handles this.

    IMPORTANT: `ReadConfig()` returns `DefaultConfig()` when file doesn't exist. `DefaultConfig().DefaultHost` is `"app.keeperhub.io"`. So `cfg.DefaultHost != ""` will be true even when no config file exists, and will return `"app.keeperhub.io"` -- same as the hardcoded default. This is benign but means the hardcoded `defaultHost` line is now only reached on `ReadConfig()` error. This is acceptable.

    Actually, reconsider: the cleaner approach is to check if config.yml actually exists and has a non-default value. But that adds complexity. Since both paths return the same value ("app.keeperhub.io"), the behavior is identical. The only functional change is: when config.yml has a CUSTOM default_host, it is now used before the hardcoded fallback. This is exactly what we want.
